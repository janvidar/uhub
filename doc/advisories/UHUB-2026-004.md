# UHUB-2026-004 — Remote heap memory corruption (code execution) in `check_user_agent()`

- **Severity:** Critical
- **Attack vector:** Network, pre-password (effectively unauthenticated), no user
  interaction
- **Impact:** Attacker-controlled heap write — remote code execution candidate;
  at minimum reliable denial of service
- **CVSS 3.1 (suggested):** 9.8 — `AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H`
- **Affected:** uhub <= 0.5.0 (and all pre-release builds before 0.6.0)
- **Fixed in:** 0.6.0 (commit `7782f7d`)

## Description

`check_user_agent()` (`src/core/inf.c`) composes a user's user-agent string from
two client-supplied INF fields — the user-agent *product* (`ADC_INF_FLAG_USER_AGENT_PRODUCT`)
and *version* (`ADC_INF_FLAG_USER_AGENT_VERSION`) — into a fixed 33-byte buffer
`user->id.user_agent` (`char[MAX_UA_LEN + 1]`, `MAX_UA_LEN == 32`) embedded in the
heap-allocated `struct hub_user`.

The vulnerable code:

```c
offset = strlen(str);                                       /* product: attacker, UNBOUNDED */
memcpy(user->id.user_agent, str, MIN(offset, MAX_UA_LEN));  /* 1st copy bounded to 32 */
...
memcpy(user->id.user_agent + offset,                        /* dest = buffer + attacker offset */
       str,                                                 /* version: attacker bytes */
       MIN(strlen(str), MAX_UA_LEN - offset));              /* size_t underflow when offset > 32 */
```

An attacker controls all three operands of the second `memcpy`:

- **Destination offset** — making the product string longer than 32 bytes sets
  `offset` to an arbitrary value, so `user_agent + offset` points well past the
  33-byte buffer.
- **Length** — `MAX_UA_LEN - offset` is computed in `size_t`; when `offset > 32`
  it underflows to a value near `2^64`, so `MIN(strlen(version), ...)` collapses
  to `strlen(version)` — fully attacker-chosen.
- **Content** — the version bytes (printable UTF-8 after ADC unescape; may include
  spaces and newlines, not NUL).

The result is a controlled-offset, controlled-length, controlled-content write
into the heap past the `hub_user` allocation — a memory-corruption primitive
suitable for overwriting adjacent heap chunk metadata or groomed pointers, i.e. a
realistic path to remote code execution. The minimum impact is a reliable crash
(full-service DoS, since uhub is single-threaded).

## Reachability

`check_user_agent` runs inside `hub_perform_login_checks` on the first `BINF`.
Both `check_logged_in` and the ADC `GPA`/`PAS` password exchange happen *after*
it, so the write fires **before any password is verified** — effectively
unauthenticated, even on a registered-only hub.

## Remediation

Upgrade to **uhub 0.6.0** or later. The fix clamps the product length with
`offset = MIN(strlen(str), MAX_UA_LEN)`, which both bounds the first copy and
keeps `MAX_UA_LEN - offset` non-negative for the version copy; the buffer is also
explicitly NUL-terminated at `user_agent[MAX_UA_LEN]`.

## Workarounds

None at the protocol layer — the corruption occurs during login INF processing
before authentication. Restrict network reach (firewall/allowlist) until upgraded.
Building with heap hardening (e.g. hardened allocator, ASan in testing) may turn
the corruption into a clean abort but does not prevent exploitation in general.
