# UHUB-2026-003 — Resource-exhaustion denial of service (fix pending in 0.6.1)

- **Severity:** Medium
- **Attack vector:** Network; connection flood is unauthenticated
- **Impact:** Denial of service (file-descriptor / memory exhaustion, search
  amplification); one path additionally allows an out-of-bounds table index
- **Status:** Fixed in `master`, **not yet in a tagged release**. **0.6.0 remains
  vulnerable.**
- **Planned fix:** 0.6.1

## Description

Several resource-exhaustion vectors remain exploitable in 0.6.0:

- **Unbounded concurrent connections** (`d5314a2`) — there is no ceiling on
  simultaneously tracked connections, so a pre-login connection flood exhausts
  file descriptors and memory. Additionally, when `RLIMIT_NOFILE` is raised above
  the network backend's `conns[]` table capacity, an accepted fd can index out of
  bounds. *Unauthenticated.*
- **Fixed per-connection send-queue limit** (`283e3ad`) — `get_max_send_queue()`
  returned a static value, letting a slow or idle client pin large per-connection
  buffers and exhaust hub memory. The fix scales the hard limit with the
  connected-user count.
- **Search amplification** (`1471dab`, issue #71) — very short substring searches
  were broadcast to every connected user with no minimum length; `limit_min_search`
  now rejects them.
- **OOM NULL-deref** in `hub_set_variables` (`ffd6070`).

## Remediation

These ship in **0.6.1**. Until then, operators on 0.6.0 should:

- enforce a connection-rate / max-connections limit at the firewall or via a
  reverse proxy,
- set a conservative `max_send_buffer`,
- and once 0.6.1 is available, configure `limit_min_search` (e.g. `3`).
