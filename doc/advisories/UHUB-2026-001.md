# UHUB-2026-001 — Unauthenticated heap out-of-bounds read in `adc_msg_parse()`

- **Severity:** High
- **Attack vector:** Network, pre-authentication, no privileges, no user interaction
- **Impact:** Denial of service (hub crash); out-of-bounds read of adjacent heap memory
- **CVSS 3.1 (suggested):** 7.5 — `AV:N/AC:L/PR:N/UI:N/S:U/C:N/I:N/A:H`
- **Affected:** uhub <= 0.5.0 (and all pre-release builds before 0.6.0)
- **Fixed in:** 0.6.0 (commit `fa0f9b5`)
- **Reference:** issue #85

## Description

The ADC message parser `adc_msg_parse()` (`src/adc/message.c`) is the first code
that runs against bytes from a connecting peer — it executes **before the `HSUP`
handshake and before any authentication**. The feature-cast loop for `F`-type
messages indexed the input buffer without bounds-checking against the message
length:

```c
n = 10;
while (line[n] == '+' || line[n] == '-')   /* no n < length guard */
{
    ...
    temp_sid[0] = line[n++];               /* reads 4 bytes, unchecked */
    temp_sid[1] = line[n++];
    ...
}
```

An unauthenticated remote attacker can send a truncated feature-broadcast line
ending on `+` or `-` (e.g. `FMSG AAAB +TCP4+`) to read up to **5 bytes past the
end of the heap allocation**. A separate `length == 0` path dereferenced
`line[0]` before any guard. Depending on heap layout and whether the hub is
built with hardening/ASan, this causes a crash and terminates the
single-threaded hub for all connected users.

## Remediation

Upgrade to **uhub 0.6.0** or later. The fix bounds the loop with `n < length`,
verifies `n + 4 <= length` before each 4-byte feature SID read, and early-returns
on `length == 0`. Regression tests cover the reported `FMSG AAAB +TCP4+` case and
a partial-SID variant.

## Workarounds

None at the protocol layer — the bug triggers before authentication. Restricting
network reach (firewall/allowlist) is the only pre-upgrade mitigation.
